# WS30 — i18n parité + chasse aux strings en dur

**Périmètre** : `src/messages/fr.json`, `src/messages/es.json`, `src/messages/en.json`, `src/i18n/routing.ts`, `src/i18n/request.ts`, `src/i18n/navigation.ts` + balayage des strings UI en dur dans `src/**/*.tsx` (hors dashboard admin `src/components/admin/dashboard/*` + `_dashboard/data.ts` = ES-en-dur intentionnel, hors corps légal `/legal/*` = exception assumée).
**Fichiers lus** : ~22 (3 fichiers de messages + 3 fichiers i18n + ~16 composants/pages inspectés pour les strings en dur) · **Lignes parcourues (approx.)** : ~7 200 (3×2173 messages + config + composants)
**Synthèse** : P0=0 · P1=1 · P2=2 · P3=4

## État de la parité (résultat des comparaisons programmatiques)

Vérifié par tokenizer/flatten sur les 3 fichiers :
- **Parité des clés : PARFAITE.** 1703 clés-feuilles identiques dans les 3 fichiers, **0 clé manquante / 0 clé en trop** dans n'importe quelle direction (fr↔es, fr↔en, es↔en).
- **0 clé dupliquée** dans chaque fichier (tokenizer sibling-scope — `JSON.parse` aurait masqué une duplication).
- **Parité des types de valeur : PARFAITE.** 0 mismatch string/array/object.
- **Parité des interpolations : PARFAITE.** 0 placeholder/argument ICU divergent (`{count}`, `{page}`, `{name}`, `{brand}`, `{slug}`, plurals ICU). Les 3 langues ont exactement les mêmes arguments par clé.
- **Valeurs vides** : une seule (`Auth.strength.empty = ""` dans les 3) — **intentionnelle** (label vide à score 0, rendu uniquement si `password.length > 0`, cf. `PasswordStrength.tsx:54`). Pas un défaut.
- **Marqueurs TODO/FIXME/lorem** : aucun (les hits `todo`/`xxx` étaient des faux positifs : « todo » espagnol + `XXX-XXXX` format téléphone présent et cohérent dans les 3).

La parité structurelle est donc **exemplaire**. Les findings ci-dessous portent sur (a) des valeurs non traduites recopiées et (b) des strings UI en dur dans le code.

## Findings

### [WS30-01] Module admin « réservations » entièrement non-localisé (Spanish en dur) — P1
- **Fichier** : `src/components/admin/reservations/FilterBar.tsx:15-21,55,57,90,102-105` · `ReservationsTable.tsx:62,76-80,114,145,179,198-199,208` · `ReservationDrawer.tsx:126,128-139,165,167,177,181,188,191,210,229,241` · `BulkActionBar.tsx:30,32,39,44` · `types.ts:45-49,73-74`
- **Catégorie** : i18n
- **Constat** : Tout le sous-système admin des réservations hardcode l'espagnol au lieu de passer par `useTranslations`. Exemples : libellés d'onglets `'Todas'/'Reservadas'/'Confirmadas'/'Entregadas'/'Expiradas'/'Canceladas'` (`FilterBar` + `types.ts`), placeholder `"Buscar por referencia, cliente, teléfono…"`, options de tri `'Más recientes'/'Más antiguas'/'Total descendente'`, en-têtes de colonnes `'Artículos'/'Total'/'Estado'/'Acciones'`, libellés de statut `'Reservada'/'Confirmada'/…`, libellés d'action `'Marcar confirmada'/'Marcar entregada'/'Marcar confirmadas'`, `'ud'/'uds'`, aria-labels (`'Seleccionar todo'`, `'Deseleccionar fila'`, `'Acciones por lote'`), boutons (`'Abrir WhatsApp'`, `'Sin acción siguiente'`, `'Cancelar reserva'`, `'Guardando…'`, `'Guardado'`), sections (`'Cliente'`, `'Nombre'`, `'Nota interna · solo equipo FARMAU'`). La page parente `src/app/admin/reservations/page.tsx` ET `NewReservationDrawer.tsx`, eux, utilisent bien `useTranslations` — et le namespace `Admin.reservations` (~27 clés) existe — mais ces 5 fichiers + `types.ts` n'y ont jamais été branchés. Aucune clé pour ces libellés (onglets/tri/colonnes/statuts) n'existe d'ailleurs dans le namespace.
- **Impact** : Contredit l'invariant projet « admin entièrement localisé FR/ES/EN ». Un admin qui bascule le panneau en FR ou EN voit toute la page réservations en espagnol (onglets, colonnes, boutons, drawer, actions par lot). Surface admin la plus opérationnelle (gestion des click-&-collect au quotidien) → impact réel pour l'usage trilingue. Non bloquant (l'espagnol reste lisible), mais c'est une régression i18n large, pas un cas isolé.
- **Reco** : Ajouter au namespace `Admin.reservations` (FR/ES/EN) les clés manquantes (tabs, sortOptions, colonnes, statusLabels, actionLabels, sections, aria) et remplacer les littéraux par `t(...)` dans les 5 composants + factoriser `STATUS_LABEL`/`nextStatusLabel`/`sortOptions` de `types.ts` en clés i18n (passer le `t` ou une map localisée). C'est un travail mécanique mais volumineux (~40 chaînes).
- **Confiance** : haute

### [WS30-02] Cart drawer (public) : compteur d'articles en espagnol en dur — P2
- **Fichier** : `src/components/cart/CartDrawerSummary.tsx:32`
- **Catégorie** : i18n
- **Constat** : `{t('subtotal')} · {itemCount} {itemCount === 1 ? 'producto' : 'productos'}`. Le mot `'producto'/'productos'` est hardcodé en espagnol dans un composant **public** (drawer panier monté depuis `NavBar` → `CartDrawer` → `CartDrawerSummary`). Une clé ICU pluralisée toute prête existe pourtant déjà : `Cart.drawerProductsCount` (`"{count, plural, =0 {0 produit} =1 {1 produit} other {# produits}}"` en FR, équivalents ES/EN).
- **Impact** : En locale FR ou EN, le sous-total du panier affiche un mot espagnol (`2 productos`) à côté d'un libellé correctement traduit (`Sous-total ·`). Visible par tout utilisateur ouvrant son panier — incohérence de langue dans un flux cœur.
- **Reco** : Remplacer par `{t('drawerProductsCount', { count: itemCount })}` (la clé existe et gère le pluriel dans les 3 langues — supprime aussi la logique de pluriel ad-hoc).
- **Confiance** : haute

### [WS30-03] aria-labels en dur (mono-langue) sur composants publics — P2
- **Fichier** : `src/components/reservation/StepIndicator.tsx:29` (`aria-label="reservation steps"`, EN) · `src/components/Breadcrumb.tsx:21` (`aria-label="Breadcrumb"`, EN)
- **Catégorie** : i18n / a11y
- **Constat** : Deux composants publics utilisés dans des flux clés (StepIndicator dans le tunnel `/reservation`, Breadcrumb dans `ProductClient.tsx`) portent un `aria-label` hardcodé en anglais. Le reste de leur contenu visible est correctement localisé (StepIndicator utilise `useTranslations('Reservation.stepIndicator')`).
- **Impact** : Les utilisateurs de lecteur d'écran en FR/ES entendent un nom de repère (landmark) en anglais. Faible nombre d'utilisateurs mais c'est précisément la population a11y. « Breadcrumb » est une convention parfois tolérée non traduite ; « reservation steps » l'est moins.
- **Reco** : Ajouter une clé (ex. `Reservation.stepIndicator.ariaLabel` — le namespace existe déjà — et un `Breadcrumb.ariaLabel` ou réutiliser un namespace `Nav`/`Common`) et passer le `t()` à l'aria-label. Pour Breadcrumb (composant générique), accepter un prop `ariaLabel` rempli par le parent localisé.
- **Confiance** : haute

### [WS30-04] `CartIcon.tsx` = code mort, contient un aria-label FR en dur — P3
- **Fichier** : `src/components/CartIcon.tsx:21` (`aria-label="Ouvrir le panier"`)
- **Catégorie** : dette / i18n
- **Constat** : `grep -rn "CartIcon" src` ne retourne **aucune** importation/utilisation hors du fichier lui-même → composant non monté nulle part. Il embarque un `aria-label` français hardcodé (et la NavBar utilise sa propre icône panier, pas celui-ci).
- **Impact** : Aucun en runtime (jamais rendu) ; dette + l'aria-label hardcodé y est moot. Signalé surtout comme code mort.
- **Reco** : Supprimer `src/components/CartIcon.tsx` (confirmer une dernière fois l'absence d'import dynamique). Si conservé, l'aria-label devra être localisé.
- **Confiance** : haute (zéro usage trouvé sur tout `src/`)

### [WS30-05] Placeholders d'exemple en dur dans des formulaires admin (incohérence) — P3
- **Fichier** : `src/app/admin/annonce/_components/BannerFormModal.tsx:206,212` (`placeholder="Dra. María Rosa Cabrera"`, `placeholder="Pharmacienne, Santo Domingo"`) · `src/app/admin/settings/page.tsx:238` (`placeholder="Lun-Vie 6h30-17h · Sáb 8h-16h"`)
- **Catégorie** : i18n
- **Constat** : Ces deux écrans admin sont par ailleurs largement localisés (`BannerFormModal` appelle `useTranslations`/`t(` ~25 fois), mais quelques `placeholder` d'exemple restent en dur (mélange FR + ES). Valeurs d'exemple, donc moins visibles que des labels.
- **Impact** : Mineur — placeholder d'exemple dans un panneau admin ; l'admin EN voit un placeholder FR/ES. Pas un blocage.
- **Reco** : Externaliser ces placeholders dans les namespaces `Admin.modals.banner` / `Admin.settings`. À traiter dans la même passe que WS30-01 (cohérence admin).
- **Confiance** : haute

### [WS30-06] 404 racine `not-found.tsx` : langue mélangée (EN + ES) en dur — P3
- **Fichier** : `src/app/not-found.tsx:15,17,23`
- **Catégorie** : i18n
- **Constat** : Le 404 racine (hors segment `[locale]`, donc i18n indisponible — documenté en commentaire) affiche un heading EN (`"Page not found"`, `"The page you're looking for…"`) **et** un CTA ES (`"Volver al inicio"`). Le `/[locale]/not-found.tsx` est bien localisé (11 `t(`) et capte 99 % des cas.
- **Impact** : Très faible (page de secours rarement atteinte : URLs ne matchant aucun segment locale). Mais le mélange EN/ES dans un même écran est négligé.
- **Reco** : Au minimum unifier la langue (tout EN, lingua franca de secours) ; les chaînes ne peuvent pas passer par next-intl ici (hors `[locale]`), donc constante figée cohérente.
- **Confiance** : haute

## Points positifs (court)
- **Parité des messages exemplaire** : 1703 clés-feuilles strictement identiques dans les 3 langues, aucune divergence de structure, de type, de placeholder, ni de doublon. Rare à ce niveau de propreté.
- `src/i18n/request.ts` : résolution duale correcte et sûre — segment `[locale]` prioritaire, fallback cookie `farmau_admin_locale` validé par `hasLocale`, fallback final `defaultLocale`. Le cookie est validé à l'écriture (Zod `setLocaleBody` dans `/api/admin/set-locale`, route `requireAdmin`) ET à la lecture. Pas de faille de fall-through pour les pages publiques.
- ICU plurals correctement utilisés (ex. `Cart.summary`, `Favoris.count`, `Account.reservations.subtitle`) avec branches `=0/=1/other` cohérentes dans les 3 langues.
- Le balayage des strings UI en dur sur **toute** la surface publique (hors admin) ne remonte qu'**un seul** nœud de texte JSX en dur (`not-found.tsx` racine, hors `[locale]` = contraint) et **un seul** ternaire de texte (`CartDrawerSummary` `producto/productos`). Le code public est quasi exhaustivement passé par `useTranslations`.
- Les valeurs « identiques entre langues » détectées sont, à une poignée près, légitimement partagées : noms propres (lieux, personnes, certifications), marque `FARMAU`, adresses, cognats (`Contact`, `Image`, `Page`), libellés de tri `A — Z`.

## Signalements hors périmètre (1 ligne chacun, max 5)
- `src/app/admin/page.tsx:147,168,184,169` — titres de sections dashboard en espagnol en dur (`Reservas e ingresos`, `Catálogo e inventario`, `Clientes y actividad`, `… productos · … marcas`) : c'est la page qui compose les widgets dashboard → relève de la **décision intentionnelle ES-en-dur** du dashboard (brief §3) ; non compté comme défaut, noté une seule fois ici.
- Corps des pages `/legal/*` (CGV/confidentialité) : EN retombe sur le contenu FR (documenté `page.tsx:30-34`) → exception assumée hors namespace ; un visiteur EN voit le juridique en français (gap connu, à valider juriste, non re-développé).
- `Reservation.shipping.zones.*` : libellés de zones de livraison présents dans les 3 langues mais relèvent de l'UI shipping orpheline (click-&-collect only, `lib/shipping.ts` hardcodé) — décision intentionnelle, pas un défaut i18n.

## Zones non couvertes / à re-vérifier humainement
- **Qualité linguistique** (au-delà de la parité de clés) non auditée mot à mot : je n'ai pas relu la justesse/idiomatisme des traductions ES/EN sur les 1703 chaînes. Quelques EN partagent l'orthographe FR `Catalogue` (vs US `Catalog`) — valide en anglais britannique, choix à trancher éditorialement, non bloquant.
- **Import dynamique de `CartIcon`** (WS30-04) : confirmé absent par grep statique ; si un chargement par chaîne dynamique existait (improbable ici), le verdict « mort » serait à revoir.
- La couverture « string en dur » repose sur des heuristiques regex/AST-light : un littéral UI inhabituel (concaténation, template complexe) pourrait échapper. Les patterns courants (texte JSX, props label/title/placeholder/alt/aria, ternaires, toast/alert) ont tous été balayés sur la surface publique.
