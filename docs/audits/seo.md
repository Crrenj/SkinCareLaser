# Audit SEO

> Audit du site e-commerce FARMAU (Next.js 15 App Router, ~353 produits, 13 marques).
> Branche `main`, date : 2026-05-19.

---

## Synthese

**Note globale : 2 / 10 (critique)**

Le site est quasiment **invisible pour Google** dans son etat actuel. Aucune metadata propre aux pages, aucun sitemap, aucun robots.txt, aucune donnee structuree, URL produits en UUID (aucun mot-cle), `<html lang="en">` alors que tout le contenu est en francais. Pour un e-commerce dermo-cosmetique cible par des recherches longues (`creme reparatrice avene`, `ducray novophane shampoing`, etc.), c'est un blocage SEO majeur.

La bonne nouvelle : la fondation technique est saine (Next.js 15 App Router, `next/image`, fonts Geist en `display: 'swap'`, headers de securite). La plupart des fixes sont des ajouts de fichiers / fonctions, sans refonte.

**Top 3 quick wins SEO :**

1. **`app/sitemap.ts` + `app/robots.ts` + `metadataBase`** (1-2 h). Indispensable pour que Google decouvre et indexe les 353 produits. Sans sitemap.xml, l'indexation depend uniquement du maillage interne, qui est faible (catalogue paginates cote client, voir #5).
2. **`generateMetadata()` sur `/product/[id]` + `/catalogue` + `/a-propos`** (2-3 h). Titres et descriptions specifiques par produit (`{nom} - {marque} | FARMAU` + extrait description), OpenGraph image = premiere image produit. Impact CTR x2 a x3 dans les SERPs et partages reseaux sociaux.
3. **Slug-based URLs `/product/[slug]`** (3-4 h, breaking). La colonne `slug` existe deja en BDD (`db/schema.sql:57`, `UNIQUE`). Remplacer `/product/{uuid}` par `/product/avene-cicalfate-creme-reparatrice` apporte des mots-cles dans l'URL, ce qui est un facteur de ranking direct + augmente le CTR. Prevoir un redirect 301 UUID -> slug pour conserver les eventuels liens existants.

Note Lighthouse SEO actuelle attendue : **65-75/100** (penalite forte sur "Document does not have a `<meta name="description">`", "Document does not have a valid `hreflang`", "Image elements do not have `[alt]` attributes" sur certaines `<img>` natives).

## Findings

### 1. Metadata generique sur layout root, aucune metadata sur pages enfants — Impact: High

**Fichier** : `src/app/layout.tsx:19-22`
**Probleme** :

```tsx
export const metadata: Metadata = {
  title: "FARMAU",
  description: "Generated by FARMAU",
};
```

- Title brut "FARMAU" sans contexte metier (catalogue, dermatologie, marques...).
- Description est le placeholder par defaut de `create-next-app` (`"Generated by FARMAU"`). Google et les LLM utilisent cette description pour decider d'afficher / referencer la page.
- Aucune metadata sur `app/page.tsx`, `app/catalogue/page.tsx`, `app/product/[id]/page.tsx`, `app/a-propos/page.tsx`, `app/contact/page.tsx` -> toutes les pages heritent du meme title/description.
- Pas de `metadataBase` -> les URLs OG seront resolues en relatif et casseront pour les crawlers (`https://yourdomain/image.png` impossible a recalculer).

**Fix** : remplacer le metadata root par un template + ajouter `metadataBase`, puis declarer une metadata locale ou `generateMetadata()` sur chaque page. Voir section "Implementations recommandees".

---

### 2. Pas de `generateMetadata()` dynamique sur la page produit — Impact: High

**Fichier** : `src/app/product/[id]/page.tsx`
**Probleme** : la page produit ne retourne aucune metadata. Pour ~353 produits, Google verra 353 pages avec exactement le meme title `FARMAU` et la meme description `Generated by FARMAU`. Cela cree un probleme classique de **contenu duplique perceptu** dans les SERPs et tue le ranking long-tail (chaque produit devrait pouvoir capturer ses propres mots-cles : `Avene Cicalfate creme reparatrice`, `Ducray Anaphase shampoing antichute`, etc.).

**Fix** : exporter `generateMetadata()` qui requete Supabase pour recuperer nom, description, marque, image, puis construit `title`, `description`, `openGraph`, `twitter`, `alternates.canonical`. Voir section "Implementations recommandees".

---

### 3. URL produit basee sur l'UUID (`/product/[id]`) au lieu du slug — Impact: High

**Fichier** : `src/app/product/[id]/page.tsx:84-102` + `src/components/ProductCard.tsx:23`
**Probleme** :

```tsx
<Link href={`/product/${product.id}`} className="block h-full group" prefetch={false}>
```

Les URLs ressemblent a `/product/8f2c7e10-9b3d-4a1f-8e6c-b21d4f9a37e0` — aucun mot-cle, ne donne aucun signal SEO, mauvais pour le CTR (l'utilisateur copie / colle un UUID dans un message vs une URL lisible). Or la colonne `slug` existe deja :

```sql
-- db/schema.sql:54-66
CREATE TABLE IF NOT EXISTS public.products (
  id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  slug TEXT UNIQUE,
  ...
);
```

Et l'admin genere bien des slugs (`src/app/admin/product/page.tsx:167`).

**Fix** :

1. Renommer le dossier `src/app/product/[id]` -> `src/app/product/[slug]`.
2. Charger le produit via `.eq('slug', slug)` au lieu de `.eq('id', id)`.
3. Modifier `ProductCard.tsx` pour utiliser `product.slug`.
4. Ajouter une redirection 301 UUID -> slug dans `next.config.ts` ou via middleware pour conserver les anciens liens.

```ts
// src/app/product/[slug]/page.tsx (extrait)
const { data: prodRaw } = await supabase
  .from('products')
  .select(PRODUCT_SELECT)
  .eq('slug', slug)
  .single<RawProduct>()
```

```tsx
// src/components/ProductCard.tsx — passer le slug au lieu de l'id
<Link href={`/product/${product.slug}`} prefetch={false}>
```

Optionnel : middleware de redirection UUID -> slug si liens externes existent deja.

### 4. Aucun sitemap.xml — Impact: High

**Fichier** : aucun (`src/app/sitemap.ts` absent, `public/sitemap.xml` absent).
**Probleme** : Google decouvre les pages d'un site soit via les sitemaps soumis dans Search Console, soit via le crawl du maillage interne. Le maillage actuel passe par `/catalogue` qui pagine cote client (`CatalogueClient.tsx:351-387`, pagination en `useState`) — les pages 2, 3, 4... ne sont jamais crawlees parce qu'elles n'existent pas dans le HTML initial. Sans sitemap, beaucoup des 353 produits sont **invisibles**.

**Fix** : creer `src/app/sitemap.ts` qui requete Supabase pour generer dynamiquement les URLs. Voir section "Implementations recommandees".

---

### 5. Aucun robots.txt — Impact: High

**Fichier** : aucun (`src/app/robots.ts` absent, `public/robots.txt` absent).
**Probleme** : sans `robots.txt`, Google crawle par defaut mais ne sait pas ou se trouve le sitemap, ni quelles routes exclure. Le `/admin/*`, `/api/*`, `/cart`, `/login`, `/signup` ne devraient pas etre crawles (et pour `/admin` c'est aussi un risque de fuite d'info — voir audit securite).

**Fix** : creer `src/app/robots.ts`. Voir section "Implementations recommandees".

---

### 6. Aucune donnee structuree (JSON-LD) — Impact: High

**Fichier** : `src/app/product/[id]/page.tsx`, `src/app/layout.tsx`, `src/app/page.tsx`
**Probleme** : aucun `<script type="application/ld+json">` n'existe nulle part. Les rich snippets Google (prix, stock, rating, breadcrumb) sont impossibles sans donnees structurees. C'est l'effort le plus rentable pour un e-commerce — un produit avec rich snippet (etoiles + prix + dispo) double souvent son CTR.

Schemas requis :
- **Organization** (sur layout) : nom, logo, URL, adresse, telephone, `sameAs` (reseaux sociaux).
- **WebSite** (sur layout) : nom, URL, `potentialAction.SearchAction`.
- **Product** (sur `/product/[slug]`) : nom, image, brand, sku, offers (prix, currency, availability).
- **BreadcrumbList** (sur `/product/[slug]` et `/catalogue`).
- **LocalBusiness** ou **Pharmacy** (sur `/a-propos`) : horaires, adresse, geo.

**Fix** : voir section "Implementations recommandees" (4 snippets prets a copier).

### 7. `<html lang="en">` alors que le contenu est en francais — Impact: Medium

**Fichier** : `src/app/layout.tsx:31`
**Probleme** :

```tsx
<html lang="en" className={`${geistSans.variable} ${geistMono.variable}`}>
```

Tout le contenu est en francais (`Accueil`, `Catalogue`, `Nos Services`, `Decouvrez notre selection`, etc.). Google detecte la mismatch et :
- penalise le ranking sur les requetes francaises ;
- desactive certaines features (auto-translate banner, voice search FR).

**Fix** : utiliser `lang="fr"` ou — mieux — preparer le futur multilingue (toggle FR/EN/ES present dans `NavBar.tsx:53-61`).

```tsx
// src/app/layout.tsx
<html lang="fr" className={`${geistSans.variable} ${geistMono.variable}`}>
```

Plus tard, si le multilingue est implemente, gerer `lang` dynamiquement via segment `[locale]` (voir #11).

### 8. Aucune balise OpenGraph / Twitter Cards — Impact: Medium

**Fichier** : `src/app/layout.tsx`, toutes les pages
**Probleme** : aucune balise `og:title`, `og:image`, `og:type`, `twitter:card`. Resultat : un partage WhatsApp / Facebook / X / LinkedIn d'un produit FARMAU affichera juste l'URL brute (`/product/8f2c-...`) sans titre ni image. C'est catastrophique pour le trafic social, sachant que la PME communique via WhatsApp Business (visible dans `a-propos/page.tsx:149`).

**Fix** : metadata template OG sur layout, override par page :

```tsx
// src/app/layout.tsx — extrait

  openGraph: {
    type: 'website',
    locale: 'fr_DO',
    siteName: 'FARMAU',
    images: [{ url: '/image/logo_trans.png', width: 800, height: 600, alt: 'FARMAU' }],
  },
  twitter: { card: 'summary_large_image', site: '@farmau' },
}
```

### 9. Aucune gestion de canonical URL — Impact: Medium

**Fichier** : toutes les pages
**Probleme** : si un produit est accessible via plusieurs URLs (UUID + slug pendant la transition, ou via query params de filtres), Google peut indexer les variantes et diluer le signal. Aucun `<link rel="canonical">` n'est present.

**Fix** : utiliser `alternates.canonical` dans les metadata Next.js :

```ts
// src/app/product/[slug]/page.tsx — dans generateMetadata
return {
  // ...
  alternates: {
    canonical: `/product/${product.slug}`,
  },
}
```

Idem pour `/catalogue` qui doit canonicaliser vers `/catalogue` (sans les query params de filtres, qui ne sont d'ailleurs pas dans l'URL aujourd'hui — autre probleme : voir #15).

### 10. Multilingue (FR/EN/ES) annonce mais non implemente — Impact: Medium

**Fichier** : `src/components/NavBar.tsx:53-61`
**Probleme** : le selecteur de langue affiche `Francais / English / Espanol` mais le `<li>` n'a aucun `onClick` et il n'existe pas de structure `[locale]` dans le router. Pour une pharmacie en Republique Dominicaine, l'absence d'espagnol fait perdre **enormement** de trafic local (l'espagnol est langue officielle). Sans `hreflang`, Google ne sait pas qu'il existe des versions par langue.

**Fix** (long terme) :

1. Adopter la structure `src/app/[locale]/...` avec `[locale]` = `fr | es | en`.
2. Utiliser `next-intl` ou `next-translate`.
3. Generer les balises `hreflang` dans la metadata :

```ts
export async function generateMetadata({ params }) {
  const { locale, slug } = await params
  return {
    alternates: {
      canonical: `/${locale}/product/${slug}`,
      languages: {
        'fr-DO': `/fr/product/${slug}`,
        'es-DO': `/es/product/${slug}`,
        'en-US': `/en/product/${slug}`,
      },
    },
  }
}
```

A defaut, retirer le toggle non-fonctionnel de la NavBar pour eviter de confondre crawlers et utilisateurs.

### 11. Variable d'environnement URL production absente — Impact: Medium

**Fichier** : `.env.local`
**Probleme** : seul `NEXT_PUBLIC_SUPABASE_*` est defini. Aucun `NEXT_PUBLIC_SITE_URL`. Cela impacte :
- `metadataBase` de Next.js — sans elle, les `og:image` resolvent en relatif et beaucoup de scrapers (LinkedIn, certains LLMs) ne les suivent pas ;
- le `sitemap.ts` et le `robots.ts` qui doivent connaitre l'URL canonique de production ;
- les liens absolus dans les emails / partages.

**Fix** : ajouter au `.env.local`, `.env.production` et a la doc :

```bash
NEXT_PUBLIC_SITE_URL=https://farmau.do
```

Puis dans le code, toujours utiliser :

```ts
const siteUrl = process.env.NEXT_PUBLIC_SITE_URL ?? 'https://farmau.do'
```

### 13. Native `<img>` au lieu de `next/image` sur la page produit — Impact: Medium

**Fichier** : `src/components/ProductClient.tsx:67, 79`
**Probleme** :

```tsx
<img
  src={product.images[0].url}
  alt={product.images[0].alt || product.name}
  className="w-full h-full object-contain"
/>
```

- Pas de dimensions explicites (`width` / `height`) -> Cumulative Layout Shift (CLS) penalise par Google.
- Pas d'optimisation auto (WebP/AVIF, responsive) malgre la config `next.config.ts:47` qui declare ces formats.
- Pas de `priority` sur l'image principale (qui est LCP candidate) -> ralentit le LCP, autre signal Core Web Vitals.

**Fix** :

```tsx
import Image from 'next/image'

<div className="aspect-square bg-gray-100 rounded-lg overflow-hidden">
  <Image
    src={product.images[0].url}
    alt={product.images[0].alt || product.name}
    width={600}
    height={600}
    className="w-full h-full object-contain"
    priority   // car LCP du produit
    sizes="(max-width: 1024px) 100vw, 50vw"
  />
</div>
```

Idem pour les vignettes secondaires (sans `priority`, avec `loading="lazy"` automatique).

### 15. Filtres catalogue cote client invisibles pour Google — Impact: Low (mais opportunite forte)

**Fichier** : `src/components/CatalogueClient.tsx:27-203`
**Probleme** : tous les filtres (marque, gamme, type de peau, besoin...) sont en `useState`. Resultat :
- L'URL ne change pas quand l'utilisateur filtre `/catalogue?brand=avene` -> impossible de partager un filtre, impossible pour Google d'indexer une page "Avene", "Anti-acne", "SPF 50+".
- Chaque marque / gamme / type est un mot-cle haute valeur (`creme avene anti-tache`, `ducray shampoing antichute`) — actuellement inacessible aux moteurs.

**Fix** (effort moyen) :

1. Passer en query params via `useSearchParams` + `useRouter().replace()`.
2. Lire les params en server component pour generer des metadata speciales : title = `Catalogue Avene | FARMAU`, description = `Decouvrez les produits Avene...`.
3. Optionnellement, creer des pages dediees `/marques/[slug]` (cf. database `brands.slug` existe deja).

```tsx
// src/app/marques/[slug]/page.tsx
export async function generateMetadata({ params }) {
  const { slug } = await params
  const supabase = await createSupabaseServerClient()
  const { data: brand } = await supabase.from('brands').select('*').eq('slug', slug).single()
  return {
    title: `${brand.name} — Produits dermo-cosmetiques`,
    description: `Decouvrez la gamme ${brand.name} disponible chez FARMAU...`,
  }
}
```

C'est un projet plus large mais aussi le plus gros levier de croissance organique (chaque marque devient une landing page).

### B. `generateMetadata()` pour la page produit

```tsx
// src/app/product/[slug]/page.tsx (extrait — apres migration UUID -> slug)
import type { Metadata } from 'next'
import { createSupabaseServerClient } from '@/lib/supabaseServer'

type Params = { slug: string }

export async function generateMetadata(
  { params }: { params: Promise<Params> }
): Promise<Metadata> {
  const { slug } = await params
  const supabase = await createSupabaseServerClient()

  const { data } = await supabase
    .from('products')
    .select(`
      name,
      description,
      slug,
      product_images ( url, alt ),
      product_ranges ( range:ranges ( name, brand:brands ( name ) ) )
    `)
    .eq('slug', slug)
    .single()

  if (!data) {
    return { title: 'Produit introuvable' }
  }

  const brand = data.product_ranges?.[0]?.range?.brand?.name ?? ''
  const image = data.product_images?.[0]?.url
  const titleBase = brand ? `${data.name} — ${brand}` : data.name
  const description =
    data.description?.slice(0, 155) ??
    `Decouvrez ${data.name}${brand ? ` de ${brand}` : ''} chez FARMAU, votre pharmacie dermatologique.`

  return {
    title: titleBase,
    description,
    alternates: {
      canonical: `/product/${data.slug}`,
    },
    openGraph: {
      type: 'website',
      title: titleBase,
      description,
      url: `/product/${data.slug}`,
      images: image
        ? [{ url: image, width: 1200, height: 1200, alt: data.name }]
        : undefined,
    },
    twitter: {
      card: 'summary_large_image',
      title: titleBase,
      description,
      images: image ? [image] : undefined,
    },
  }
}
```

### E. `app/sitemap.ts` — sitemap dynamique

```ts
// src/app/sitemap.ts
import type { MetadataRoute } from 'next'
import { createSupabaseServerClient } from '@/lib/supabaseServer'

const SITE_URL = process.env.NEXT_PUBLIC_SITE_URL ?? 'https://farmau.do'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const supabase = await createSupabaseServerClient()

  const [products, brands] = await Promise.all([
    supabase.from('products').select('slug, updated_at').eq('is_active', true),
    supabase.from('brands').select('slug'),
  ])

  const staticRoutes: MetadataRoute.Sitemap = [
    { url: `${SITE_URL}/`, changeFrequency: 'weekly', priority: 1.0 },
    { url: `${SITE_URL}/catalogue`, changeFrequency: 'daily', priority: 0.9 },
    { url: `${SITE_URL}/a-propos`, changeFrequency: 'monthly', priority: 0.5 },
    { url: `${SITE_URL}/contact`, changeFrequency: 'monthly', priority: 0.5 },
  ]

  const productRoutes: MetadataRoute.Sitemap = (products.data ?? [])
    .filter(p => p.slug)
    .map(p => ({
      url: `${SITE_URL}/product/${p.slug}`,
      lastModified: p.updated_at ? new Date(p.updated_at) : undefined,
      changeFrequency: 'weekly',
      priority: 0.8,
    }))

  const brandRoutes: MetadataRoute.Sitemap = (brands.data ?? []).map(b => ({
    url: `${SITE_URL}/marques/${b.slug}`,
    changeFrequency: 'weekly',
    priority: 0.7,
  }))

  return [...staticRoutes, ...productRoutes, ...brandRoutes]
}
```

## Recommandations prioritaires

Ordre d'execution recommande (par ratio impact / effort) :

1. **[1-2 h] Bases techniques** : `metadataBase` + `lang="fr"` + `.env NEXT_PUBLIC_SITE_URL` + `app/robots.ts` + `app/sitemap.ts`. Sans ces 5 elements, le reste n'a pas d'effet (le sitemap est ce qui declenche l'indexation des 353 produits).
2. **[2 h] Metadata statiques** sur `page.tsx`, `catalogue/page.tsx`, `a-propos/page.tsx`, `contact/page.tsx` (snippet G). Effet immediat sur les SERPs des 4 pages principales.
3. **[3 h] `generateMetadata()` produit + JSON-LD Product + Breadcrumb** (snippets B + C). Multiplie le potentiel de trafic long-tail par ~10 (353 pages produits indexables individuellement avec rich snippets).
4. **[1 h] JSON-LD Organization + WebSite** dans le layout (snippet D). Permet le knowledge panel Google + searchbox sitelink.
5. **[3-4 h] Migration UUID -> slug** : renommer `[id]` -> `[slug]`, mise a jour `ProductCard`, ajout middleware de redirect 301. Critique pour le SEO long-terme.
6. **[1 h] H1 sur home + catalogue** + corriger les `<img>` natives en `next/image` dans `ProductClient.tsx`.
7. **[2 h] Convertir le logo en WebP** + verifier toutes les images du dossier `/public/image/`.
8. **[chantier] Filtres catalogue en query params** (`?brand=avene&type=anti-age`). Permettra de partager les filtres + creera plus de pages indexables.
9. **[chantier] Pages marque dediees** `/marques/[slug]` (utilise `brands.slug` deja en BDD). Equivalent landing page par marque, levier organique majeur.
10. **[plus tard] Multilingue FR/ES/EN** avec `[locale]` + `hreflang`. Lever le toggle non-fonctionnel de NavBar dans l'intervalle.

Apres ces 9 points, attendre une note Lighthouse SEO de **95-100/100** et une indexation complete des 353 produits dans Google Search Console dans les 4-6 semaines suivantes.
